# CHIEF Data Model — Events

This document describes the Event data model implemented in Sprint 4 (Orbit).

## Event Fields

- id (UUID): Primary identifier for the event, generated by the server.
- title (string): Short, human-readable title for the event (max 255 characters).
- description (text, optional): Longer description or summary of the event.
- start_datetime (ISO 8601 datetime): When the event starts. Stored as naive datetime (UTC assumed by convention).
- end_datetime (ISO 8601 datetime): When the event ends.
- event_type (string): One of: "meeting", "reminder", "birthday", "personal". Used for UI styling.
- priority (integer): Numeric priority; higher values indicate higher priority. Default 0.
- location (string, optional): Short location string (max 255 chars).
- notes (text, optional): Additional notes.
- created_at (datetime): Timestamp when the event was created.
- updated_at (datetime): Timestamp when the event was last updated.

## Purpose

The Event model represents calendar items displayed in the CHIEF dashboard. It is intentionally small and focused on the data required to show events in the calendar and today panel.

## API Endpoints

- GET /events
  - Returns list of events ordered by start_datetime.
  - Response: 200 OK, JSON array of Event objects.

- GET /events/{id}
  - Returns a single event by id.
  - Response: 200 OK with Event, or 404 Not Found.

- POST /events
  - Create a new event.
  - Request body: Event fields (title, start_datetime, end_datetime, event_type, etc.).
  - Response: 201 Created with new Event object.

- PUT /events/{id}
  - Update an existing event.
  - Request body: Event fields.
  - Response: 200 OK with updated Event, or 404 Not Found.

- DELETE /events/{id}
  - Delete an event.
  - Response: 204 No Content, or 404 Not Found.

## Event Bus Messages

Changes to events publish messages on the core EventBus (in-memory):

- event_created — payload: the created Event object as JSON
- event_updated — payload: the updated Event object as JSON
- event_deleted — payload: { "id": "<event-id>" }

## Limitations

- No recurrence support.
- No attendee lists.
- No reminders or notifications.
- No timezone handling — datetimes are naive and handled as-is.
- No authentication or authorization.

This is intentionally minimal; future sprints will expand functionality following CHIEF's architecture and design principles.
